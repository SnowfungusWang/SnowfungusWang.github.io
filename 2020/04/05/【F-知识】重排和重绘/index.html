【F-知识】重排和重绘

Posted on 2020-04-05 Edited on 2020-04-08

亲娘嘞，影响性能呀！【邢育森皱眉.jpg】

“重绘”不一定需要”重排”，“重排”必然导致”重绘”。

是什么？

网页生成的过程

  1. HTML代码转化成DOM
  2. CSS代码转化成CSSOM（CSS Object Model）
  3. 结合DOM和CSSOM，生成一棵渲染树（包含每个节点的视觉信息）
  4. 生成布局（flow）：将所有渲染树的所有节点进行平面合成
  5. 绘制（paint）：将布局绘制（paint）在屏幕上

前面3步相对快，4、5比较耗时，加起来叫渲染。（怪不得vue把dom移走劫持一下不是很慢）

网页生成的时候，至少会渲染一次。用户访问的过程中，还会不断重新渲染。

重排 & 重绘

重新渲染，就需要重新生成布局和重新绘制。前者叫做”重排”（reflow），后者叫做”重绘”（repaint）。

可能触发：

  • 修改DOM
  • 修改样式表
  • 用户事件（比如鼠标悬停、页面滚动、输入框键入文字、改变窗口大小等等）

性能调优

影响

  1. 浏览器会尽量把样式排在一起一次性操作完了

  2. 从性能角度考虑，尽量不要把读操作和写操作，放在一个语句里面。

    • 就是应用上面的变动达成下一条语句，就是写完立刻读，那就得先完成一次重排重绘

  3. 样式的写操作之后，如果有下面这些属性的读操作，都会引发浏览器立即重新渲染。

    • offsetTop/offsetLeft/offsetWidth/offsetHeight
    • scrollTop/scrollLeft/scrollWidth/scrollHeight
    • clientTop/clientLeft/clientWidth/clientHeight
    • getComputedStyle()

  4. 一个讨论：仅读取会重绘吗

  5. 一般的规则：

    • 样式表越简单，重排和重绘就越快。
    • 重排和重绘的DOM元素层级越高，成本就越高。
    • table元素的重排和重绘成本，要高于div元素

触发

  1. 重排：几何因素。原理是一样的，都为求一个“即时性”和“准确性”

    • 页面第一次渲染：在页面发生首次渲染的时候，所有组件都要进行首次布局，这是开销最大的一次重排。

    • 浏览器窗口尺寸改变

    • 元素位置和尺寸发生改变的时候

    • 新增和删除可见元素

    • 内容发生改变（文字数量或图片大小等等）

    • 元素字体大小变化。

    • 激活CSS伪类（例如：:hover）。

    • 设置style属性 9.查询某些属性或调用某些方法。比如说：

      offsetTop、offsetLeft、 offsetWidth、offsetHeight、scrollTop、scrollLeft、scrollWidth、scrollHeight、clientTop、clientLeft、clientWidth、clientHeight

    • 当调用getComputedStyle方法，或者IE里的currentStyle时，也会触发重排。

  2. 重绘：

    • vidibility、outline、背景色等属性的改变
    • 重排之后

调优

  1. DOM 的多个读操作（或多个写操作），应该放在一起。不要两个读操作之间，加入一个写操作。

  2. 第二条，如果某个样式是通过重排得到的，那么最好缓存结果。避免下一次用到的时候，浏览器又要重排。

  3. 第三条，不要一条条地改变样式，而要通过改变class，或者csstext属性，一次性地改变样式。

    • csstext：DOM元素的一个属性对象，批量内联属性
    • 改变csstest会覆盖之前的×（毕竟是内联样式了）
    • 优先级高于单独的style.xxx

  4. 【？？？？？】尽量使用离线DOM，而不是真实的网面DOM，来改变元素样式。比如，操作Document Fragment对象，完成后再把这个对象加入DOM。再比如，使用 cloneNode() 方法，在克隆的节点上进行操作，然后再用克隆的节点替换原始节点。

    • Document Fragment：文档片段接口。表示一个没有父级文件的最小文档对象。作为一个轻量版的 Document，用于存储已排好版的或尚未打理好格式的 XML 片段。

    • 感觉上是一个DOM的改动先搞个副本，全整完再替换进去

    • 说白了就是偷懒，你也不知道哪些语句或者顺序会引起重绘重排，就干脆虚拟操作（移除，不触发）

  5. 先将元素设为display: none（需要1次重排和重绘），然后对这个节点进行100次操作，最后再恢复显示（需要1次重排和重绘）。这样一来，你就用两次重新渲染，取代了可能高达100次的重新渲染。

    • 少换多
    • 也有点移除，不触发那味儿

  6. position属性为absolute或fixed的元素，重排的开销会比较小，因为不用考虑它对其他元素的影响。

    • 和那个DOM层级高开销大有异曲同工之妙

  7. 只在必要的时候，才将元素的display属性为可见，因为不可见的元素不影响重排和重绘。另外，visibility : hidden的元素只对重绘有影响，不影响重排。

    • 还是那句老话：移除，不触发

  8. 【？？？】使用虚拟DOM的脚本库，比如React等。

    • vue呢？

  9. 使用window.requestAnimationFrame()、window.requestIdleCallback() 这两个方法调节重新渲染。

    • RAF！似李！！

  10. GPU 硬件加速：应用 GPU 的图形性能对浏览器中的一些图形操作交给 GPU 来完成，因为 GPU 是专门为处理图形而设计，所以它在速度和能耗上更有效率。

    • Canvas2D，布局合成
    • CSS3转换（transitions）
    • CSS3 3D变换（transforms）
    • WebGL和视频(video)。

刷新率与帧（动画）

刷新率

  1. 网页动画的每一帧都是重新渲染，相当于重新生成一张图片。
  2. 刷新率：一秒之间能够完成多少次重新渲染。英文为FPS（frame per second）。
    • 数值越大帧率越高越清晰
    • 60次重新渲染，就是60FPS
    • 一般要30~60FPS才会比较流畅
  3. 大多数显示器都是60HZ，为了与系统一致，以及节省电力，浏览器会自动按照这个频率，刷新动画（如果可以做到的话）。
    • 一秒之内进行60次重新渲染，每次重新渲染的时间不能超过16.66毫秒。
  4. 想达到60帧的刷新率，就意味着JavaScript线程每个任务的耗时，必须少于16毫秒。一个解决办法是使用Web Worker，主线程只用于UI渲染，然后跟UI渲染不相干的任务，都放在Worker线程。

window.requestAnimationFrame()

  1. 调节重新渲染，其中的代码会放到下一次重新渲染时执行

  2. 可以使用window.requestAnimationFrame()，让读操作和写操作分离，把所有的写操作放到下一次重新渲染。

  3. 例子

    • 页面滚动事件（scroll）的监听函数

    • 网页动画

    • 旋转

window.requestIdleCallback()

  1. 指定只有当一帧的末尾有空闲时间，才会执行回调函数。
    • 只有当前帧的运行时间小于16.66ms时，回调函数才会执行。否则，就推迟到下一帧，如果下一帧也没有空闲时间，就推迟到下下一帧，以此类推。
  2. 还可以接受第二个参数，表示指定的毫秒数。如果在指定 的这段时间之内，每一帧都没有空闲时间，那回调函数将会强制执行。
    • 最迟xx毫秒之后执行
  3. 回调函数可以接受deadline对象作为参数，有一个方法一个属性
    • timeRemaining() 方法：返回当前帧还剩余的毫秒。
      • 这个方法只能读，不能写，而且会动态更新。
      • 可以不断检查这个属性，如果还有剩余时间的话，就不断执行某些任务。一旦这个属性等于0，就把任务分配到下一轮
    • didTimeout属性：返回一个布尔值，表示指定的时间是否过期。
      • 如果回调函数由于指定时间过期而触发，那么
        • timeRemaining方法返回0
        • didTimeout 属性等于 true
